# 功能模块描述模板

> 每个功能模块开发前，需先填写本文档，作为开发和 AI 测试的依据。
> 开发者只需填写占位符内容，AI 会读取此模板结合实际业务自动生成完整文档。

---

## 格式约束（AI 生成时必须遵守）

> 以下规则确保测试脚本能准确解析文档内容，生成时必须严格遵守。

1. **章节编号固定**：`1-9` 章节的编号和标题不可更改，不可增减章节
2. **流程标题格式**：必须是 `#### 流程N：名称`（N 从1开始，中文冒号）
3. **流程内容格式**：必须包裹在代码块 ` ``` ` 中，包含 `前置：`、`步骤：`、`最终验证：` 三个固定标记
4. **步骤格式**：每步以 `  序号. [操作类型] 描述 → 期望结果`，操作类型包括：`[新增]` `[查询]` `[编辑]` `[删除]` `[验证]` `[操作]`
5. **表格格式**：所有表格必须包含表头行和分隔行 `| --- |`
6. **依赖关系**：无依赖时填写"无"，不要留空
7. **文件命名**：模块文件名使用英文短横线命名，如 `user-management.md`

## 6. 测试用例（流程驱动）

> 测试思路：按业务场景走流程，每步只判断成功/失败，
> 上一步的结果作为下一步的输入，失败时才分析具体原因。

> **适用场景说明**：
>
> | 场景类型            | 是否需要数据隔离测试 | 示例                             | 常见 Bug                       |
> | ------------------- | :------------------: | -------------------------------- | ------------------------------ |
> | **分类/类型下操作** |       ✅ 需要        | 字典管理：在"用户状态"下新增数据 | 新增变成编辑，覆盖其他类型数据 |
> | **有层级结构**      |       ✅ 需要        | 部门管理：在"研发部"下新增员工   | 员工被添加到错误的部门         |
> | **有归属关系**      |       ✅ 需要        | 课程管理：为"数学课"添加章节     | 章节被添加到错误的课程         |
> | **平铺增删改查**    |      ❌ 不需要       | 优惠券管理：直接创建优惠券       | 无                             |
> | **单一类型资源**    |      ❌ 不需要       | 用户管理：创建用户               | 无                             |
>
> **数据隔离测试的常见风险点**：
>
> - **参数污染**：表单数据混入了分类/类型对象的字段（如 id, name, code）
> - **误判模式**：新增时因为有 id 字段被误判为编辑，覆盖了其他数据
> - **类型错乱**：由于参数错误，数据被添加到错误的分类中
>
> **当需要数据隔离测试时，请遵守以下规则**：
>
> 1. **明确选择**：必须写 `[选择] 选择XX分类/类型`
> 2. **反向验证**：不仅要验证"包含"，还要验证"不包含在其他分类"
> 3. **交叉验证**：验证操作只影响了目标对象，未影响其他对象
> 4. **快照对比**：操作前后对比其他分类的数据，确认未被修改
>
> **示例（需要数据隔离的场景）**：
>
> ```
> 步骤：
>   1. [选择] 选择"用户状态"类型
>   2. [记录] 记录"订单状态"的当前数据快照
>   3. [新增] 在"用户状态"下创建数据 → 期望成功
>   4. [验证] 查询"用户状态" → 期望包含新增数据
>   5. [验证] 查询"订单状态" → 期望不包含新增数据
>   6. [验证] 对比"订单状态"快照 → 期望未被修改
> ```
>
> **示例（不需要数据隔离的场景）**：
>
> ```
> 步骤：
>   1. [新增] 创建优惠券 → 期望成功
>   2. [验证] 查询优惠券列表 → 期望包含新增数据
>   3. [删除] 删除该优惠券 → 期望成功
>   4. [验证] 查询列表 → 期望不包含该优惠券
> ```

### 6.1 流程测试

#### 流程1：完整生命周期（增删改查）

```
前置：无

步骤：
  1. [新增] 创建一条数据 → 期望成功，记录ID
  2. [查询] 查询该数据 → 期望存在
  3. [编辑] 修改该数据 → 期望成功
  4. [验证] 查询该数据 → 期望已更新
  5. [删除] 删除该数据 → 期望成功
  6. [验证] 查询该数据 → 期望不存在

最终验证：数据已被完全清除
```

#### 流程2：异常拦截

```
前置：已存在至少一条数据

步骤：
  1. [新增] 使用重复/无效数据创建 → 期望失败，提示具体原因
  2. [新增] 使用空必填项创建 → 期望失败，提示具体原因
  3. [删除] 删除被引用的数据 → 期望失败，提示已被使用
  4. [验证] 原数据未被影响

最终验证：所有异常被正确拦截，数据无变化
```

#### 流程3：关联操作（如模块间有数据交互）

```
前置：关联模块数据已就绪

步骤：
  1. [操作] 触发关联操作 → 期望成功
  2. [操作] 基于上一步结果执行 → 期望成功
  3. [验证] 关联数据状态正确

最终验证：关联数据一致性正常
```
