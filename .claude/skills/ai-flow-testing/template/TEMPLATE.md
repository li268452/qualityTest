# 功能模块描述模板

> 每个功能模块开发前，需先填写本文档，作为开发和 AI 测试的依据。
> 开发者只需填写占位符内容，AI 会读取此模板结合实际业务自动生成完整文档。

---

## 格式约束（AI 生成时必须遵守）

> 以下规则确保测试脚本能准确解析文档内容，生成时必须严格遵守。

1. **章节编号固定**：`1-9` 章节的编号和标题不可更改，不可增减章节
2. **流程标题格式**：必须是 `#### 流程N：名称`（N 从1开始，中文冒号）
3. **流程内容格式**：必须包裹在代码块 ` ``` ` 中，包含 `前置：`、`步骤：`、`最终验证：` 三个固定标记
4. **步骤格式**：每步以 `  序号. [操作类型] 描述 → 期望结果`，操作类型包括：`[新增]` `[查询]` `[编辑]` `[删除]` `[验证]` `[操作]`
5. **表格格式**：所有表格必须包含表头行和分隔行 `| --- |`
6. **依赖关系**：无依赖时填写"无"，不要留空
7. **文件命名**：模块文件名使用英文短横线命名，如 `user-management.md`

## 0. 功能流程描述

> 本章节使用自然语言描述功能流程，AI 测试脚本会自动解析并生成测试用例。
>
> **重要**：只需描述"做什么"，无需关心"怎么做"（技术实现）。

### 流程描述格式

每个功能流程按以下格式描述：

```
### 场景名称：[场景描述]

**涉及接口**：[接口列表，可选]

**触发条件**：
- [条件1]
- [条件2]

**操作流程**：
1. [操作步骤1]
2. [操作步骤2]
3. [操作步骤3]

**完成标志**：[如何判断流程完成]
```

### 操作类型关键词

描述操作流程时，使用以下关键词，AI 脚本会自动识别：

| 关键词 | 说明          | 示例                |
| ------ | ------------- | ------------------- |
| 打开   | 进入页面/弹窗 | 打开新增弹窗        |
| 填写   | 输入数据      | 填写名称="测试数据" |
| 选择   | 从列表中选择  | 选择类型="普通类型" |
| 点击   | 触发按钮操作  | 点击保存按钮        |
| 调用   | 后端接口      | 调用创建接口        |
| 返回   | 接口响应      | 返回成功            |
| 刷新   | 更新页面数据  | 刷新列表            |
| 跳转   | 页面跳转      | 跳转到详情页        |

#### 示例2：启用数据流程

```
### 场景名称：启用数据

**操作流程**：
1. 选择要启用的数据

### 自动生成规则

AI 脚本会根据场景描述自动生成测试用例：

**规则1：操作类型映射**

- `填写/输入` → `[新增]`
- `打开/进入` → `[查询]`（如果是进入列表页）
- `点击保存/提交` → `[新增]`（如果有填写步骤）
- `选择/勾选` → `[选择]`
- `刷新/加载` → `[验证]`

**规则2：参数提取**

- 从 `填写 字段名="值"` 中提取参数
- 从 `选择 字段名="值"` 中提取参数
- 参数会自动映射到实际的字段名（通过字段分析）

**规则3：流程结构生成**

```
#### 流程N：[场景名称]

前置：[触发条件]

步骤：
  [根据操作流程自动生成]

最终验证：[完成标志]
```

### 编写注意事项

1. **使用标准动词**：尽量使用上述关键词中的动词
2. **参数格式**：使用 `字段名="值"` 或 `字段名=值` 格式
3. **场景粒度**：一个场景对应一个完整的业务流程
4. **异常处理**：异常场景请直接在第6节"测试用例"中人工编写

### 4.2 Mock API 返回格式规范（必填）

> **重要**：所有 Mock API 必须遵循统一的返回格式规范，否则测试脚本会报错。

**标准返回格式**：

```javascript
// 成功
{
  "code": 200,          // 业务状态码（必填，数字类型）
  "data": {...},        // 返回数据（必填，成功时为对象，失败时为 null）
  "msg": "操作成功"     // 提示信息（必填，字符串类型）
}

// 失败
{
  "code": 400,          // 业务状态码（必填，数字类型）
  "data": null,         // 返回数据（必填，失败时必须为 null）
  "msg": "错误原因"     // 错误提示（必填，字符串类型）
}
```

**分页数据格式**：

```javascript
{
  "code": 200,
  "data": {
    "records": [...],   // 数据列表（必填，数组类型）
    "total": 100,       // 总记录数（必填，数字类型）
    "page": 1,          // 当前页码（必填，数字类型）
    "size": 10          // 每页大小（必填，数字类型）
  },
  "msg": "查询成功"
}
```

**状态码规范**：

| Code | 说明         | 使用场景                     | 示例               |
| ---- | ------------ | ---------------------------- | ------------------ |
| 200  | 操作成功     | 所有成功操作                 | 新增成功、修改成功 |
| 400  | 请求参数错误 | 参数缺失、格式错误、校验失败 | 名称不能为空       |
| 404  | 资源不存在   | 数据不存在                   | 数据不存在         |
| 409  | 资源冲突     | 数据重复、状态冲突           | 名称已存在         |
| 422  | 数据验证失败 | 业务规则验证失败             | 数值必须大于0      |

**错误提示规范**：

| 规范           | 示例                        |
| -------------- | --------------------------- |
| 明确指出问题   | ✅ "名称不能为空"           |

| 包含上下文     | ✅ "名称「测试数据」已存在" |
| 避免模糊提示   | ❌ "参数错误"               |
| 避免无意义提示 | ❌ "操作失败"               |

## 6. 测试用例（流程驱动）

> 测试思路：按业务场景走流程，每步只判断成功/失败，
> 上一步的结果作为下一步的输入，失败时才分析具体原因。

> **步骤描述规范（重要）**：
>
> 为了让通用测试脚本能正确解析和执行，请遵守以下规范：
>
> **1. 参数描述格式**：
>
> - ✅ 正确：`名称="测试数据", 编码="test_code", 数量=10`
> - ❌ 错误：`名称为测试数据且编码为test_code`（测试脚本无法解析）
> - ✅ 正确：`标签="正常", 值="1", 排序=1`
> - ❌ 错误：`标签是正常且值是1`（测试脚本无法解析）
>
> **2. 异常测试参数描述**：
>
> - ✅ 正确：`创建用户，名称="" → 期望失败`（明确指定空字符串）
> - ❌ 错误：`使用空名称创建用户 → 期望失败`（测试脚本不知道什么是"空名称"）
> - ✅ 正确：`创建订单，数量=-1 → 期望失败`（明确指定负数）
> - ✅ 正确：`创建字典，编码="existing_code" → 期望失败`（明确指定重复值）
> - ❌ 错误：`创建订单，数量为负 → 期望失败`（测试脚本无法解析"负"）
>
> **3. 操作类型关键词**：
>
> - 新增/创建：`[新增]` `[创建]`
> - 查询/验证：`[查询]` `[验证]`
> - 编辑/修改：`[编辑]` `[修改]`
> - 删除：`[删除]`
> - 启用/禁用：`[启用]` `[禁用]` `[关闭]`
> - 选择/切换：`[选择]` `[切换]` `[下拉]`
>
> **4. 期望结果格式**：
>
> - 必须包含 `→ 期望结果`
> - 失败时必须写明原因：`→ 期望失败，提示具体原因`
> - 成功时可省略：`→ 期望成功`

> **适用场景说明**：
>
> | 场景类型            | 是否需要数据隔离测试 | 示例                             | 常见 Bug                       |
> | ------------------- | :------------------: | -------------------------------- | ------------------------------ |
> | **分类/类型下操作** |       ✅ 需要        | 字典管理：在"用户状态"下新增数据 | 新增变成编辑，覆盖其他类型数据 |
> | **有层级结构**      |       ✅ 需要        | 部门管理：在"研发部"下新增员工   | 员工被添加到错误的部门         |
> | **有归属关系**      |       ✅ 需要        | 课程管理：为"数学课"添加章节     | 章节被添加到错误的课程         |
> | **平铺增删改查**    |      ❌ 不需要       | 数据管理：直接创建数据           | 无                             |
> | **单一类型资源**    |      ❌ 不需要       | 用户管理：创建用户               | 无                             |
>
> **数据隔离测试的常见风险点**：
>
> - **参数污染**：表单数据混入了分类/类型对象的字段（如 id, name, code）
> - **误判模式**：新增时因为有 id 字段被误判为编辑，覆盖了其他数据
> - **类型错乱**：由于参数错误，数据被添加到错误的分类中
>
> **当需要数据隔离测试时，请遵守以下规则**：
>
> 1. **明确选择**：必须写 `[选择] 选择XX分类/类型`
> 2. **反向验证**：不仅要验证"包含"，还要验证"不包含在其他分类"
> 3. **交叉验证**：验证操作只影响了目标对象，未影响其他对象
> 4. **快照对比**：操作前后对比其他分类的数据，确认未被修改
>
> **示例（需要数据隔离的场景）**：
>
> ```
> 步骤：
>   1. [选择] 选择"用户状态"类型
>   2. [记录] 记录"订单状态"的当前数据快照
>   3. [新增] 在"用户状态"下创建数据，标签="正常", 值="1", 排序=1 → 期望成功
>   4. [验证] 查询"用户状态" → 期望包含新增数据
>   5. [验证] 查询"订单状态" → 期望不包含新增数据
>   6. [验证] 对比"订单状态"快照 → 期望未被修改
> ```
>
> **示例（不需要数据隔离的场景）**：
>
> ```
> 步骤：
>   1. [新增] 创建数据，名称="测试数据", 数量=20 → 期望成功
>   2. [验证] 查询数据列表 → 期望包含新增数据
>   3. [删除] 删除该数据 → 期望成功
>   4. [验证] 查询列表 → 期望不包含该数据
> ```
>
> **示例（异常拦截场景）**：
>
> ```
> 步骤：
>   1. [新增] 创建用户，名称="" → 期望失败，提示名称不能为空
>   2. [新增] 创建用户，名称="重复名称" → 期望失败，提示名称已存在
>   3. [新增] 创建订单，数量=-1 → 期望失败，提示数量必须大于0
>   4. [验证] 验证原数据未被影响 → 期望数据未被修改
> ```
