## 2. 功能概述

字典管理用于维护系统中的下拉选项、枚举值等数据字典。支持字典类型的新增、编辑、删除，以及字典数据的新增、编辑、删除和排序。是系统中其他模块（如用户管理、订单管理等）的基础支撑模块。

## 3. 使用场景

### 场景1：管理员新增字典类型

- **角色**：系统管理员
- **前置条件**：已登录系统，拥有字典管理权限
- **操作步骤**：
  1. 进入字典管理页面
  2. 点击"新增字典类型"按钮
  3. 填写字典名称（如"用户状态"）、字典编码（如"user_status"）、备注
  4. 点击保存
- **预期结果**：字典类型列表中新增一条记录，状态为启用

### 场景2：管理员维护字典数据

- **角色**：系统管理员
- **前置条件**：已存在字典类型
- **操作步骤**：
  1. 在字典类型列表中点击某个字典类型
  2. 右侧展示该类型下的字典数据列表
  3. 点击"新增字典数据"
  4. 填写字典标签（如"正常"）、字典值（如"1"）、排序号
  5. 点击保存
- **预期结果**：字典数据列表中新增一条记录

### 场景3：其他模块引用字典

- **角色**：开发人员（通过接口调用）
- **前置条件**：字典类型和字典数据已维护
- **操作步骤**：
  1. 通过字典编码调用接口获取字典数据
  2. 将返回数据渲染为下拉框/标签等
- **预期结果**：正确显示字典数据列表

### 场景4：管理员删除字典

- **角色**：系统管理员
- **前置条件**：已存在字典数据
- **操作步骤**：
  1. 选中要删除的字典数据
  2. 点击删除按钮
  3. 确认删除
- **预期结果**：数据被删除，列表刷新

## 4. 接口设计

| 接口         | 方法   | 路径                             | 说明                            |
| ------------ | ------ | -------------------------------- | ------------------------------- |
| 字典类型列表 | GET    | /api/dict/types                  | 分页查询，支持名称/编码模糊搜索 |
| 新增字典类型 | POST   | /api/dict/types                  | 新增字典类型                    |
| 编辑字典类型 | PUT    | /api/dict/types/{id}             | 修改字典类型                    |
| 删除字典类型 | DELETE | /api/dict/types/{id}             | 删除字典类型（级联删除数据）    |
| 字典数据列表 | GET    | /api/dict/types/{typeCode}/items | 查询某类型下的字典数据          |
| 新增字典数据 | POST   | /api/dict/types/{typeCode}/items | 新增字典数据                    |
| 编辑字典数据 | PUT    | /api/dict/items/{id}             | 修改字典数据                    |
| 删除字典数据 | DELETE | /api/dict/items/{id}             | 删除字典数据                    |
| 根据编码查询 | GET    | /api/dict/{typeCode}             | 根据字典编码获取所有字典数据    |

### 6.1 流程测试

#### 流程1：字典类型完整生命周期

```
前置：无

步骤：
  1. [新增] 创建字典类型，名称="用户状态", 编码="user_status" → 期望成功，记录类型ID
  2. [查询] 查询该类型 → 期望存在，状态为启用
  3. [编辑] 修改名称为"用户状态V2" → 期望成功
  4. [验证] 查询该类型 → 期望名称已更新
  5. [删除] 删除该字典类型 → 期望成功
  6. [验证] 查询该类型 → 期望不存在

最终验证：该类型已完全清除
```

#### 流程2：字典数据维护流程

```
前置：已存在字典类型"user_status"

步骤：
  1. [新增] 创建字典数据，标签="启用", 值="1", 排序=1 → 期望成功
  2. [新增] 创建字典数据，标签="禁用", 值="0", 排序=2 → 期望成功
  3. [验证] 查询 user_status 下所有数据 → 期望2条，按排序升序
  4. [编辑] 修改"启用"的排序号为2 → 期望成功
  5. [验证] 查询列表 → 期望"禁用"排在前面
  6. [删除] 删除"禁用" → 期望成功
  7. [验证] 查询列表 → 期望仅剩"启用"

最终验证：列表中只有一条"启用"数据
```

#### 流程3：跨模块引用流程

```
前置：已存在字典类型"order_status"，含数据"待支付/已支付/已取消"

步骤：
  1. [调用] 通过编码查询 order_status → 期望成功
  2. [验证] 返回数据包含3个选项
  3. [禁用] 禁用字典类型 order_status → 期望成功
  4. [调用] 再次查询 order_status → 期望返回空或标记为禁用

最终验证：禁用后接口不再返回该字典数据
```

#### 流程4：异常拦截流程

```
前置：已存在字典类型"user_status"

步骤：
  1. [新增] 使用编码"user_status"再创建一个 → 期望失败，提示编码已存在
  2. [新增] 使用空名称创建字典类型 → 期望失败，提示名称不能为空
  3. [新增] 同类型下创建重复值的字典数据 → 期望失败，提示字典值已存在
  4. [验证] 删除字典类型后，关联数据一同被清除
  5. [验证] 原有数据均未被影响

最终验证：所有异常被正确拦截，数据无变化
```

### 6.2 状态检查点

| 检查点             | 检查方式           | 期望状态             |
| ------------------ | ------------------ | -------------------- |
| 新增类型后列表可见 | 查询类型列表       | 包含新增的类型       |
| 新增数据后列表刷新 | 查询类型下数据列表 | 包含新增的数据       |
| 编辑后数据已更新   | 查询详情           | 字段值与修改一致     |
| 删除后数据消失     | 查询列表           | 不包含已删除的数据   |
| 禁用后接口不返回   | 按编码查询         | 不返回禁用类型的数据 |
| 排序生效           | 查询数据列表       | 按排序号升序排列     |

## 7. 业务规则

1. 字典编码唯一，不可重复
2. 同一字典类型下，字典值不可重复
3. 删除字典类型时级联删除该类型下所有字典数据
4. 字典类型可启用/禁用，禁用后接口不返回该字典数据
5. 字典数据支持排序，按排序号升序展示
6. 字典标签和值均为必填项
