# AI 流程驱动测试任务

## 模块：dict-management

## 路径：dict-management

---

## 测试角色

你是一名 QA 测试工程师，负责对「dict-management」模块进行自动化业务流程测试。

## 测试原则

1. **流程驱动**：按业务场景依次执行操作，上一步成功才继续下一步
2. **只判成败**：每步操作只判断成功或失败，不逐字段检查返回内容
3. **链式传递**：上一步的返回数据（如ID）作为下一步的输入
4. **失败才分析**：成功就继续，失败时才抓取详细错误信息并分析原因
5. **数据一致性**：流程结束后验证最终数据状态是否正确

---

## 前置依赖

本模块依赖以下模块，测试前需确保这些模块的数据或接口可用：

- **用户管理**：获取用户状态、性别等字典数据
- **订单管理**：获取订单状态、支付方式等字典数据

## 业务规则

- 1. 字典编码唯一，不可重复
- 2. 同一字典类型下，字典值不可重复
- 3. 字典类型被引用时（被其他模块使用），不可删除
- 4. 字典类型可启用/禁用，禁用后接口不返回该字典数据
- 5. 字典数据支持排序，按排序号升序展示
- 6. 字典标签和值均为必填项

## 测试流程

### 流程1：字典类型完整生命周期

**前置条件**：无

**执行步骤**：

```
1. [新增] 创建字典类型，名称="用户状态", 编码="user_status" → 期望成功，记录类型ID
2. [查询] 查询该类型 → 期望存在，状态为启用
3. [编辑] 修改名称为"用户状态V2" → 期望成功
4. [验证] 查询该类型 → 期望名称已更新
5. [删除] 删除该字典类型 → 期望成功
6. [验证] 查询该类型 → 期望不存在
```

**最终验证**：该类型已完全清除

---

### 流程2：字典数据维护流程

**前置条件**：已存在字典类型"user_status"

**执行步骤**：

```
1. [新增] 创建字典数据，标签="启用", 值="1", 排序=1 → 期望成功
2. [新增] 创建字典数据，标签="禁用", 值="0", 排序=2 → 期望成功
3. [验证] 查询 user_status 下所有数据 → 期望2条，按排序升序
4. [编辑] 修改"启用"的排序号为2 → 期望成功
5. [验证] 查询列表 → 期望"禁用"排在前面
6. [删除] 删除"禁用" → 期望成功
7. [验证] 查询列表 → 期望仅剩"启用"
```

**最终验证**：列表中只有一条"启用"数据

---

### 流程3：跨模块引用流程

**前置条件**：已存在字典类型"order_status"，含数据"待支付/已支付/已取消"

**执行步骤**：

```
1. [调用] 通过编码查询 order_status → 期望成功
2. [验证] 返回数据包含3个选项
3. [禁用] 禁用字典类型 order_status → 期望成功
4. [调用] 再次查询 order_status → 期望返回空或标记为禁用
```

**最终验证**：禁用后接口不再返回该字典数据

---

### 流程4：异常拦截流程

**前置条件**：已存在字典类型"user_status"

**执行步骤**：

```
1. [新增] 使用编码"user_status"再创建一个 → 期望失败，提示编码已存在
2. [新增] 使用空名称创建字典类型 → 期望失败，提示名称不能为空
3. [新增] 同类型下创建重复值的字典数据 → 期望失败，提示字典值已存在
4. [删除] 删除被其他模块引用的字典类型 → 期望失败，提示已被使用
5. [验证] 原有数据均未被影响
无上游 → 字典管理 → 用户管理、订单管理等所有业务模块
```

**最终验证**：所有异常被正确拦截，数据无变化

## 状态检查点

流程执行过程中，在对应节点验证以下状态：

- **新增类型后列表可见**：通过「查询类型列表」验证 → 期望「包含新增的类型」
- **新增数据后列表刷新**：通过「查询类型下数据列表」验证 → 期望「包含新增的数据」
- **编辑后数据已更新**：通过「查询详情」验证 → 期望「字段值与修改一致」
- **删除后数据消失**：通过「查询列表」验证 → 期望「不包含已删除的数据」
- **禁用后接口不返回**：通过「按编码查询」验证 → 期望「不返回禁用类型的数据」
- **排序生效**：通过「查询数据列表」验证 → 期望「按排序号升序排列」
